# Roadmap

Статусы: ✅ done · 🚧 in progress · ⬜ not started

## Phase 0 — Настройка проекта ✅

- Angular 22 (standalone, zoneless), Node 24 LTS
- Angular Material + CDK
- Vitest (unit), Playwright (e2e)
- ESLint (`@angular-eslint`) + Prettier
- Git-репозиторий, docs/SPEC.md, docs/ROADMAP.md, CLAUDE.md

## Phase 1 — Доменная модель и хранение ✅

- TypeScript-типы из SPEC.md (`Quiz`, `Question`, `Option`, `QuizAttempt`)
- `QuizRepository` — интерфейс + localStorage-реализация
- `QuizStore` (сервис на signals) — CRUD опросников в памяти + синхронизация с репозиторием
- Unit-тесты доменной логики (создание id, валидация, CRUD)

## Phase 2 — Список опросников и базовый конструктор ✅

- Страница со списком опросников (создать/переименовать/дублировать/удалить)
- Страница редактирования опросника: список вопросов, добавление/удаление
- Сортировка вопросов через CDK Drag&Drop

## Phase 3 — Редакторы типов вопросов ✅

- Компонент-редактор для single-choice
- Компонент-редактор для multiple-choice
- Компонент-редактор для text
- Компонент-редактор для image-choice (URL картинки на старте, загрузка файлов — опционально)
- Пометка правильных ответов (для graded-режима)

## Phase 4 — Прохождение опросника (respondent mode) ✅

- Страница прохождения: рендер вопросов по типу, валидация обязательных полей
- Сохранение `QuizAttempt` в репозиторий
- Экран результата (для graded — показать баллы)

## Phase 5 — Аутентификация и роли ✅

- Страница логина (демо-доступ без бэкенда, роль admin/респондент)
- Сессия сохраняется в localStorage, кнопка «Выйти», отображение роли
- Управление опросником (создание/переименование/дублирование/удаление,
  редактор вопросов) доступно только admin — скрыто в UI и защищено
  route guard'ом
- Чтение из `AttemptRepository` (`getByQuizId`) + экран «Результаты» для
  admin — список попыток и баллов по каждому опроснику

## Phase 6 — Респонденты и статистика ✅

- `QuizAttempt.respondentName` (необязательное имя, пусто → «Аноним»),
  поле «Ваше имя» на странице прохождения
- Экран «Результаты»: колонка «Респондент» в таблице попыток, разворот
  строки — все ответы респондента по каждому вопросу
- Доменная логика подсчёта % верных/неверных ответов по каждому
  вопросу (для graded-опросников)
- Визуализация статистики по вопросам — простой bar-chart на CSS, без
  сторонней графической библиотеки

## Phase 7 — Импорт/экспорт и полировка ✅

- Экспорт опросника в JSON-файл, импорт из файла (с новым id, копией
  в список опросников)
- Предпросмотр опросника из режима редактирования — встроенная панель,
  отражающая текущий черновик (в т.ч. несохранённые правки), ответы
  в предпросмотре не сохраняются
- Пустые состояния, базовая адаптивность (перенос строк на узких
  экранах, прокручиваемая таблица результатов), сообщения об ошибках
  ввода (пустое название опросника, некорректный файл импорта)

## Backlog / после MVP

Кандидаты в будущие фазы, сгруппированные по теме — каждая со своим статусом.
Порядок не фиксирован — конкретную разбивку и приоритет уточняем перед
стартом каждого блока.

## Phase 8 — Доп. типы вопросов: простые ✅

- true-false (да/нет — частный случай single-choice со своим UX)
- dropdown (выпадающий список — альтернатива radio-списку при большом
  числе вариантов)
- number (числовой ответ с валидацией диапазона)
- date (дата/время как ответ)

## Phase 9 — Доп. типы вопросов: шкалы и веса ✅

- rating (шкала оценки — звёзды или 1–5/1–10, включая NPS 0–10)
- slider (ползунок с числовым/процентным значением)
- constant-sum (распределение 100 баллов/очков между вариантами)

## Phase 10 — Доп. типы вопросов: сложная интерактивность ✅

- word-choice (выбор/расстановка слов)
- fill-in-the-blank (текст с пропусками, респондент вставляет слова)
- ranking (сортировка вариантов, drag&drop)
- matching (сопоставление пар)
- matrix (таблица вопрос×шкала)

## Phase 11 — Доп. типы вопросов: продвинутые/нишевые ✅

- hotspot (клик по нужной области на изображении)
- file-upload (респондент прикладывает файл/фото как ответ)

## Phase 12 — Логика прохождения ✅

- Реализовать `shuffleQuestions` (перемешивание вопросов) — поле в модели
  уже есть, но нигде не используется
- Ограничение времени на попытку (таймер)
- Лимит числа попыток на респондента
- Условная логика/переходы между вопросами (skip logic)
- Страницы опросника: группировка вопросов по темам (`Quiz.pages`,
  `question.pageId`), прохождение постранично с кнопками «Далее»/«Назад»
  и валидацией по странице

## Phase 13 — Публикация и доступ ✅

- Состояние «черновик/опубликовано» (`settings.published`, по умолчанию
  `true` — старые опросники остаются доступны) — админ снимает галочку
  «Опубликован», после этого `/quiz/:id/run` показывает respondent'у
  сообщение вместо формы, а сам опросник пропадает из списка для
  non-admin (у админа остаётся, с бейджем «Черновик»)
- Защита опросника паролем/кодом доступа (`settings.accessPassword`) —
  respondent видит форму с кодом до формы прохождения
- Срок действия ссылки (`settings.expiresAt`) — после дедлайна
  прохождение блокируется сообщением об истёкшем сроке
- Предпросмотр из редактора игнорирует все три ограничения

## Phase 14 — Результаты и экспорт ✅

- Экспорт результатов прохождений в CSV (кнопка «Экспорт в CSV» на
  странице результатов, с BOM для корректного открытия в Excel)
- Фильтр по имени респондента и сортировка по дате/баллам на странице
  результатов (клик по заголовку столбца)
- Снэпшот опросника на момент прохождения (`QuizAttempt.quizSnapshot`) —
  баллы, статистика и ответы в детализации попытки теперь считаются по
  той версии вопросов, которую респондент реально видел, а не по текущей
  (правка правильного ответа задним числом больше не меняет результаты
  старых попыток); для попыток без снэпшота (сохранённых до этой фичи)
  используется текущая версия опросника как раньше
- Процент завершения (completion rate) — отложено в Phase 20, так как
  требует сохранения незавершённых попыток, которого пока нет

## Phase 15 — Backend и инфраструктура 🚧

- ✅ Backend API + серверное хранение (замена localStorage-репозитория) —
  ASP.NET Core (.NET 10, Minimal APIs) + EF Core/PostgreSQL, полностью
  нормализованная реляционная схема (`server/QuizBuilder.Api`), локальный
  запуск через `dotnet run`; фронтенд полностью переведён на HTTP-репозитории
  (`Http*Repository`), `localStorage` для данных больше не используется
- ✅ Полноценная аутентификация (замена демо-логина admin/admin) — один
  admin-аккаунт в БД (пароль хранится как PBKDF2-хэш), сессия — httpOnly cookie
  вместо флага в localStorage
- ⬜ Вебхуки/уведомления при прохождении опросника (email, Slack, Zapier-style) —
  последний пункт фазы, ещё не сделан

## Phase 16 — UX/полировка ⬜

- i18n (мультиязычный интерфейс — архитектура не должна это блокировать)
- Тёмная тема
- Расширенный a11y-аудит (клавиатурная навигация, screen reader)

## Phase 17 — CI/инфраструктура ⬜

- Расширить GitHub Actions workflow на полный `npm run verify`
  (format:check, lint, unit-тесты, build), а не только e2e
- Бейдж статуса сборки в README

## Phase 18 — Редактор: качество жизни ⬜

- Undo/redo при редактировании опросника
- Дублирование отдельного вопроса (сейчас можно дублировать только весь
  опросник)
- Автосохранение черновика (сейчас только предупреждение при закрытии
  вкладки, сохранение вручную по кнопке)
- Обложка/картинка опросника
- Кастомизация темы/брендинга опросника (логотип, цвета, фон — не
  только глобальная тёмная тема из Phase 16)

## Phase 19 — Список опросников: при масштабе ⬜

- Поиск/фильтр опросников по названию
- Сортировка списка (по дате создания/изменения, по алфавиту)

## Phase 20 — Прохождение опросника: UX ⬜

- Прогресс-бар / «Вопрос X из Y» (для непостраничных опросников — для
  постраничных уже есть «Страница X из Y», см. Phase 12)
- Сохранение недопройденной попытки (resume later)
- Настраиваемое сообщение благодарности / редирект после прохождения

## Phase 21 — Шаблоны опросников ⬜

- Галерея готовых шаблонов (типовые опросники разных категорий) для
  быстрого старта вместо пустого опросника
- Сохранение своего опросника как шаблона для повторного использования

## Phase 22 — Персонализированные результаты (outcome-квизы) ⬜

- Результат зависит от комбинации ответов, а не только от суммы баллов
  (формат «какой ты X» / маркетинговый квиз)
- Конструктор «исходов»: текст/картинка, которые показываются в
  зависимости от выбранных вариантов

## Phase 23 — Live: realtime-инфраструктура (stretch) ⬜

- WebSocket-канал поверх backend из Phase 15 — самый большой скачок
  сложности в бэклоге, браться в последнюю очередь
- Понятие live-сессии/комнаты на сервере (кто ведущий, кто участники,
  в каком опроснике)

## Phase 24 — Живой групповой режим (stretch) ⬜

- Ведущий запускает опросник, участники подключаются по PIN-коду
  (Kahoot-style), вопросы синхронно показываются всем
- Зависит от Phase 23 (realtime-канал) — без неё эту фазу не начать

## Phase 25 — Современный визуальный дизайн ⬜

- Кастомная Material-тема (цвета, типографика, отступы) вместо дефолтной
  Material Design — своя визуальная айдентика приложения
- Анимации/переходы (между вопросами при прохождении, состояния
  сохранения, hover/focus) вместо мгновенных переключений
- Иллюстрации/иконки для пустых состояний и ключевых экранов
- Общий visual polish по всем страницам, единый стиль карточек, кнопок,
  форм — отличается от Phase 18 (там брендинг конкретного опросника,
  здесь — дизайн самого приложения)

## Phase 26 — AI-проверка загруженных файлов (stretch, в самом конце) ⬜

- Для вопросов типа file-upload — проверка через vision-модель (Claude/OpenAI/
  Google Vision), соответствует ли загруженная картинка тексту вопроса
  (например, «прикрепите скан паспорта» → загружен скан паспорта, а не
  случайное фото)
- Требует бэкенд-прокси к API с ключом (нельзя вызывать платный vision-API
  напрямую из браузера) — зависит от Phase 15
- Осознанно отложено в самый конец бэклога: пользователь решил сначала
  закрыть весь остальной список фаз

## Принцип продвижения по фазам

Каждая фаза завершается рабочим, протестированным срезом функциональности
(build зелёный, unit + относящиеся e2e тесты проходят) перед переходом к
следующей. Порядок фаз может быть скорректирован по ходу работы — это рабочий
план, а не фиксированный контракт.
